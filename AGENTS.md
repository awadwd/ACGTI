# ACGTI AGENT 工作规范

本文件适用于仓库根目录及其所有子目录。除非用户明确给出更高优先级指令，否则在本仓库内工作的 Agent 必须遵守以下规则。

## 1. 项目定义

### 1.1. ACGTI 是什么

`ACGTI` = `ACG Type Indicator`

它是以现实人格鉴定工具 MBTI 为基础的“套壳”实现。本工具将经典的十六型人格分别映射到二次元语境下的核心角色形象。它测定的是受试者在现实世界中的人格倾向，并通过展示具有相应特质的二次元“人设原型”来呈现结果。

### 1.2. 产品定位

本项目默认按以下产品定位推进：

- `ACGTI` 本质上是一个被精美包装的 `MBTI` 套壳，作为现实人格鉴定工具的二次元化表达。
- `ACGTI` 的核心输出是“标准的四字母标签”，并附带与之对应的二次元角色原型解析、图片展示。
- `ACGTI` 的交互体验和视觉风格应最大程度地复刻经典的十六型人格测试网站（如 16personalities），以扁平化、清晰直观的测试落地页风格呈现。
- `ACGTI` 的价值主要来自于页面设计的还原度、逻辑严密的多维度题目，以及结果展示部分的二次元文案与立绘。

## 2. 默认工作模式

收到用户任务后，必须按以下顺序推进，不允许跳过“先理解再实现”的过程：

1. 接收用户指令
2. 阅读现有代码、文档、素材和配置
3. 形成实现方案
4. 执行实现
5. 自我验证
6. 交付成果

执行要求：

- 必须先理解仓库现状，再进行修改；即使仓库刚初始化，也要先确认目录结构与已有文件。
- 必须尽量形成完整闭环，不要只停留在建议层面。
- 应优先自主完成可落地工作，仅在缺少关键输入且无法合理推断时再询问用户。
- 完成后必须进行最基本的自我验证；若因为环境或项目阶段限制无法验证，必须明确说明。

## 3. 内容与产品表达要求

- 用户可见文案需将四字母 MBTI 类型代码作为核心分类标签进行展示。
- 采用严肃心理测试加二次元角色解析的模式，题目、选项以及结果展示参照 16personalities 的严肃风格。
- 结果内容主要包含：“人格类型（四字母代码）”、对应二次元角色的解析、雷达图/柱状图展示各个维度的倾向比例。
- 所有四字母编码必须对外作为首要识别符展示，并关联相应的二次元人设原型。

## 4. 技术实现边界

- 默认将本项目视为静态网站或前端优先项目处理。
- 若无明确需求，不主动引入后端服务、用户系统、数据库、鉴权、消息队列或云函数。
- 若需保存测试结果，优先考虑本地存储、纯前端状态或静态可部署方案。
- 任何技术选型都应服务于页面体验、内容组织和可维护性，而不是炫技。
- 新增依赖前必须考虑是否真的必要，避免在项目早期引入过重工具链。

## 5. 视觉与交互要求

- 页面设计风格要求极度趋近或复刻 16personalities 测试网站（包含扁平化 UI、经典 SaaS 风格）。
- 卡片排版、圆角、绿色/青色按钮配色和结构化进度条设计都需要向 16personalities 标准靠拢。
- 结果页应展示类似于 16personalities 的页面头部大色块、对应的二次元图片及四字母人格和称号组合。
- 移动端体验不可忽略，所有的选择器应当还原 16personalities 那种逐渐变化大小的圆圈单选框。

## 6. 代码与文件组织要求

- 保持良好的模块拆分，避免把页面逻辑、题库数据、结果映射和样式全部堆在一个文件里。
- 题目数据、类型定义、结果描述、视觉资源引用应尽量分层组织。
- 在复杂映射逻辑、评分逻辑、结果归类逻辑处补充简洁注释，说明设计意图。
- 优先写清晰、可维护的实现，不要为了“智能感”牺牲可读性。

## 7. Git 与提交要求

- 每完成一个功能，就应进行一次本地 Git 提交。
- 提交前必须先检查工作区，只提交与当前任务直接相关的文件。
- 提交信息必须使用中文，并采用“标题 + 空行 + 正文”的形式。
- 提交前应用 `git status` 和 `git diff --staged` 复核内容。
- 绝对禁止执行 `git push`。

推荐格式：

```text
<type>(<scope>): <中文摘要>

背景：
- 说明为什么要做这次改动。

实现：
- 说明改了什么、采用了什么策略。

验证：
- 说明做了哪些检查、构建或人工验证。

影响：
- 说明影响范围、限制和后续事项。
```

## 8. 安全与边界要求

- 绝不在仓库中硬编码 API Key、令牌、密码或其他敏感信息。
- 不要把 `ACGTI` 描述成心理诊断、医学评估或可替代专业意见的工具。
- 若用户提出会显著偏离本项目娱乐定位的设计，应先指出风险，再执行用户明确确认后的方案。

## 9. 实施原则

- 先读后改：先理解当前实现和项目方向，再动手。
- 内容优先：优先保证世界观、题目设计、结果表达和页面体验的一致性。
- 轻量优先：没有充分理由时，不把简单问题升级成复杂架构。
- 结果导向：交付时明确说明做了什么、怎么验证、还有什么限制。
- 尊重现状：不得随意覆盖或回退用户已有改动。
